You are building a multi-tenant Hospital CRM (India-first, simple UI). Implement the following 3 changes end-to-end (DB + API + UI + validations + audit logs). ========================================
A) DUPLICATE LEAD VALIDATION (Mobile Number)
========================================
Goal: Prevent creation of multiple ACTIVE leads with the same mobile number within the same tenant.

Definition:
- “Duplicate” means same `mobile_number` under same `tenant_id` where lead status is NOT in: Closed, Converted, Rejected.
- Mobile is stored normalized: only digits, trimmed, country code optional but consistent. Create a normalization helper.

Requirements:
1) On Lead Create:
   - If duplicate found, block creation.
   - Show an error banner/modal: "Lead already exists with this mobile number."
   - Show summary of existing lead: Lead Name, Created Date, Assigned To, Current Status.
   - Provide actions: [Open Existing Lead] [Cancel].

2) On Lead Edit:
   - If mobile number changed, re-run the same validation and block if duplicate.

Database:
- Add a DB-level constraint if feasible:
  - Unique index on (tenant_id, mobile_normalized) ONLY for active leads OR implement via application validation if partial index not supported.
- Add field: `mobile_normalized`.

API:
- Add endpoint or reuse search:
  - GET /leads/check-duplicate?mobile=... returns {isDuplicate, leadId, leadSummary}
- Ensure validation also happens inside POST/PUT lead routes.

API:

========================================
C) NEGOTIATION DISCOUNT SECTION + APPROVAL
========================================
Goal: Capture discount properly with approval status and approver.

Add a “Negotiation Discount” section in Episode Billing / Quotation screen with fields:
- notes (mandatory)
- original_quoted_amount (currency, mandatory)
- discount_percent (decimal, optional)
- discount_amount (currency, optional)
- final_amount (currency, auto-calculated, read-only)
- approved (Yes/No) default No
- approved_by (user) auto-filled when approved
- approved_at (datetime) auto-filled when approved

Rules:
1) User enters either discount_percent OR discount_amount (not both editable simultaneously).
2) Auto-calculate:
   - If percent entered: discount_amount = original * percent; final = original - discount_amount
   - If amount entered: percent = discount_amount / original; final = original - discount_amount
3) notes is mandatory whenever discount is applied (>0).
4) Approval:
   - Discount is NOT considered final unless approved = Yes.
   - When approved toggled to Yes, capture approved_by = current user and approved_at = now.
   - Only authorized roles can approve (create a config list or use roles like Center Head / Finance Head / Director).
   - Once approved, discount fields become locked (read-only). If changes needed, must “Revoke Approval” (sets approved back to No, clears approved_by/approved_at, and requires re-approval).
5) Audit log:
   - Every discount create/edit/approval/revoke must be logged (who, when, old/new values).

Database:
- Add fields to episodes (or episode_billing):
  - original_quoted_amount, discount_percent, discount_amount, final_amount,
  - discount_notes, discount_approved, discount_approved_by, discount_approved_at,
  - discount_status (Draft/Pending/Approved/Rejected optional).
- Create table: discount_audit_logs
  - id, tenant_id, episode_id, action (CREATE/UPDATE/APPROVE/REVOKE),
  - old_values_json, new_values_json, user_id, timestamp, reason(optional).

UI:

- Mobile: accordion/collapsible section.
- Validation messages should be simple and in plain English.
