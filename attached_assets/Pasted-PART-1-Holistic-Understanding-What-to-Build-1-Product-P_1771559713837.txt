PART 1: Holistic Understanding (What to Build)
1. Product Purpose
Build a multi-tenant Hospital CRM focused on Growth and Coordination: Lead capture to Appointment to Consultation Done to Conversion to Retention and Referrals.
HMS remains the system of record for clinical and billing. CRM stores only lightweight marketing and coordination data.
Hub-Spoke model: Head Office (Hub) and multiple Centres (Spokes).
Every interaction follows a strict pattern: Activity Log (what was done) + Next Action Task (what must happen next).
2. Non-Functional Goals
Responsive UI across Desktop, Tablets, and Mobile Handsets.
Multi-tenant data isolation enforced in every database table and every query.
Masters-driven system to prevent duplicate spellings (e.g., Bangalore/Bengaluru). Free-text must be minimized.
Auditability: key actions and edits logged with who/when/what changed.
Integrations ready: Telephony (tool-agnostic, Callyzer first), WhatsApp messaging, Marketing lead sources, HMS light sync.
3. Core User Roles (Tenant Scoped)
Admin (Tenant Admin)
Head Office Marketing
Central Telecalling
Branch Front Office
Branch Manager
Doctor
Doctor Assistant
Care Coordinator / Physiotherapy Team (optional for v1)
Analyst / Reporting
Platform Super Admin (optional, internal only, cross-tenant)
4. Core Modules
Masters (Admin controlled, bulk upload everywhere)
User Management (org chart + access scopes + PHI levels)
Lead Intake (all sources auto-ingest into Raw Lead Captured)
Lead Workspace (Activity Log + Next Action Task)
Appointments (OPD scheduling, doctor availability, token on check-in)
Campaigns + Attribution (UTMs, platform, landing pages)
Dashboards (Lead Gen, Conversion, ROI, Branch, SLA)
Governance (SLA rules, reminder policies, data retention, audit logs)
Customer Care (Provision only: ticket tables + create-ticket button; advanced automation deferred)
5. Funnel Truth (Single Master)
There must be ONE master controlling funnel stages: LeadStatus. Remove/avoid competing stage masters.
Raw Lead Captured
Qualified
Contacted
Appointment Booked
Reminder Running
Consultation Done (Business Achieved)
Closed Won
Closed Lost
Unqualified
Nurture
6. Stage Jump Rules
Allow Raw Lead Captured -> Closed Lost (LostReason mandatory).
Allow Raw Lead Captured -> Unqualified (Reason mandatory).
Block Raw Lead Captured -> Closed Won. Closed Won must come after Consultation Done or explicit Admin override (optional).
Enable Nurture workflows from Closed Lost / Unqualified / Dormant if AllowNurtureOption is true.
PART 2: Step-by-Step Build Instructions (Build Section by Section)
Follow this build order exactly. Do not proceed to later sections until earlier sections are implemented and validated.
GLOBAL RULES AND INSTRUCTIONS (Apply Everywhere)
Multi-tenant: Every table MUST include TenantID. Every API and query MUST filter by TenantID from session context.
Tenant resolution: Primary by subdomain. Fallback by email domain mapping or user record TenantID. No tenant selector for normal users.
Master-driven dropdowns: All dropdown values MUST come from master tables. Avoid free-text for city/area/department/status/source.
Bulk upload: Every master must support import (CSV/XLSX), template download, preview, dedupe, and import logs.
Unique constraints: Code fields unique; Name uniqueness case-insensitive within Tenant (or globally for global masters). Normalize text (trim, collapse spaces).
Audit logs: Record create/update on critical entities (Leads, Tasks, Appointments, Masters, User access changes, stage changes, assignments).
Activity + Next Action: Any lead interaction must create an ActivityLog entry. Next Action Task is mandatory except terminal statuses (Closed Won/Lost/Unqualified).
Handover acceptance: When ownership moves across departments/teams, acceptance is mandatory and must be 2-click simple (Accept/Reject).
PHI governance: CRM stores minimal non-clinical data. Restrict sensitive fields by PHIAccessLevel and role permissions.
Integrations: Implement connectors as configurable modules per tenant with encrypted secrets and sync logs.

STEP 4: Build User Management (Org Chart + Access Control)
Create a CRMUser table with: Name, Email, Phones (E164 model), Role (SystemRole), Department, Branch, Designation, EmploymentType, ReportingTo, AccessScopeType, PHIAccessLevel, IsActive.
Implement login/auth and session-based permissions.
Implement record visibility rules: Branch scope + role + assignments.
Admin screen: create/edit users, bulk upload users, assign calling lines, and view org chart (ReportingTo).
STEP 5: Build Core Transaction Tables (Database Schema Level)
Create these transaction entities with TenantID, audit fields, and proper relationships.
5A. CRM Patient and Contacts (Lightweight)
CRMPatient: minimal marketing identity; includes Primary and Secondary phone (E164), area/city, nearest branch mapping, consent flags.
Contact: family/guardian/caregiver; E164 phone model.
PatientContactLink: many-to-many with relationship and primary contact flag.
HMS Link fields on CRMPatient: HMSSystemCode, HMSPatientID, HMSLinkStatus, HMSLastSyncedAt.
5B. Leads
Lead: always created for each inquiry; links to CRMPatient and primary contact when available.
Attribution fields on lead: LeadSourceID, CampaignID, UTM fields, adset/ad/keyword/form/landing page.
Routing fields: OwnerUserID, ResponsibleUserID, AssignedBranchID, TreatmentSubDepartmentID, PreferredDoctorID (optional).
Stage fields: LeadStatusID, LastActivityAt, NextActionAt.
5C. ActivityLog and Tasks (Strict Pattern)
ActivityLog: LeadID, ActivityType, Description, Outcome, StatusChanged, OldStatus, NewStatus, DoneBy, Timestamp, Attachment(optional).
Task: RelatedEntityType (Lead/Appointment/Episode/Ticket), RelatedEntityID, TaskType, DueDateTime, AssignedTo, Status, EscalationLevel, Notes.
Rule: Completing an activity must create the next task unless LeadStatus.RequiresNextTask is false.
5D. Appointments
Appointment: LeadID, CRMPatientID, BranchID, DoctorID, ConsultationTypeID, Start/End DateTime, Status, TokenNumber, LateStatus.
Doctor availability: derived from OPDTiming + exceptions; prevent overlapping bookings.
5E. Episodes (Light)
Episode: created on Consultation Done; links to CRMPatient, Lead, Branch, Doctor, TreatmentSubDepartment; minimal non-clinical summary.
5F. Campaigns (Transaction Object)
Campaign: Platform, Channel, Name, Dates, Budget optional, Target Branch optional, LandingPageURL.
5G. Audit and Logs
AuditLog: EntityType, EntityID, Action, OldValue, NewValue, ChangedBy, Timestamp, TenantID.
BulkImportLog: EntityName, FileName, Mode (Skip/Update), counts, errors, TenantID.
ConnectorSyncLog: ConnectorType, RunAt, counts, error summary, TenantID.
STEP 6: Lead Intake Automations (All Sources -> Raw Lead Captured)
Build inbound lead endpoints/connectors for: Meta, Google, Website/Landing Pages, Google Forms, Camps.
Every inbound lead must create a Lead in LeadStatus = Raw Lead Captured.
Dedup rule: match CRMPatient by PhoneE164 (primary then secondary). If none, create minimal patient.
Auto-assign Owner and Responsible using AssignmentRules (Round Robin or least-loaded).
Auto-create SLA task: First Contact Call due in 15 minutes (configurable per tenant).
STEP 7: Lead Workspace UI (The Main Screen)
Build the Lead Workspace with these panels, all responsive:
Header summary: Patient/lead identity, phone(s), source/campaign, current status, owner/responsible, branch, speciality.
Activity Log panel: quick add activity with activity type, description, outcome, optional status change.
Next Action panel: action type, due datetime, assigned to (auto suggested), save creates Task.
Timeline: list of activities and tasks (past and upcoming).
Quick actions: Call, WhatsApp, SMS, Email, Book Appointment, Mark Lost/Unqualified (with reasons).
Guardrails: status change must validate required fields from LeadStatus.RequiredFieldsJSON.
STEP 8: Handover Acceptance (Department/Team Shift)
Add fields on Lead: CurrentDepartmentID, HandoverToUserID, HandoverStatus (None/Pending/Accepted/Rejected), RequestedAt, AcceptedAt, RejectionReason.
Trigger handover when status transition changes owner role/department.
New owner sees two buttons only: Accept or Reject.
Until accepted, lead remains with old owner; new owner sees it in Pending Acceptance list.
Create SLA for acceptance and escalate to reporting manager if breached.
STEP 9: Appointments Scheduling
Create Doctor availability calendar view from OPDTiming and exceptions.
Prevent double booking for same doctor and time overlap.
On check-in, generate TokenNumber sequential per Doctor per Day per Branch.
On appointment completion, move lead status to Consultation Done (Business Achieved) and create Episode.
STEP 10: WhatsApp Messaging Integration
Per-tenant WhatsApp connector configuration (encrypted credentials).
Consent gate: send only if WhatsApp consent is true; else fallback to SMS/call task.
All outbound/inbound messages must be logged to CommunicationLog with delivery status when available.
Use templates with variables: PatientName, DateTime, Branch, Doctor.
STEP 11: Telephony Integration (Callyzer First, Tool-Agnostic)
Implement TelephonyConnector config per tenant (Callyzer first).
Ingest call logs (includes country code and number); compute PhoneE164; map to lead/patient.
Auto-create ActivityLog entry for phone calls.
Provision outbound push: push leads/users to telephony tool using country code and national number fields where supported.
Maintain TelephonyCallLog table for raw call analytics and dedupe.
STEP 12: HMS Integration (Light)
Per-tenant HMS connector config.
Sync OPD visit list: create Raw Lead Captured leads from OPD when appropriate.
Sync consultation completion: update appointment/lead status to Consultation Done.
Store HMSPatientID on CRMPatient and resolve conflicts with manual admin mapping flow.
STEP 13: Dashboards and Reports (Explicit)
Marketing Dashboard: Raw leads by source/platform/campaign/UTM; trends; CPL optional.
Conversion Dashboard: stage-to-stage conversion %, time in stage, lost reason distribution.
Campaign ROI Dashboard: Cost per Consultation Done, Cost per Appointment if budget entered.
Branch Dashboard: appointments booked/attended, no-show %, consultation done, SLA compliance.
Telecalling Dashboard: my queue, SLA due, overdue, productivity.
SLA Compliance Dashboard: breaches by role/branch/stage.
STEP 14: Customer Care Backend (Provision Only)
Create SupportTicket and TicketActivityLog tables and minimal masters for later.
Add button on Patient/Lead screen: Create Support Ticket (simple modal).
Defer full ticket SLA automation and analytics to Phase 2.
STEP 15: Data Retention and Archival
Implement DataRetentionPolicy per tenant and entity type.
Archive old leads/episodes based on policy; keep audit trails intact.
